# SSH 作用与配置说明

## 为什么要写这份说明

在 GitHub 协作开发中，大家经常会看到两种仓库地址：

1. HTTPS
2. SSH

例如同一个仓库，可能会有下面两种写法：

### HTTPS 地址

```text
https://github.com/sysuwilliam/NC-Current-Source.git
```

### SSH 地址

```text
git@github.com:sysuwilliam/NC-Current-Source.git
```

很多刚接触 Git 的同学会疑惑：

- SSH 是干什么的？
- 为什么有些人推荐用 SSH？
- 配 SSH 会不会很麻烦？
- 我到底该用 HTTPS 还是 SSH？

这份文档就是专门解释这些问题的。

---

## SSH 到底是什么

SSH 的全称是：

`Secure Shell`

它本来是一种安全连接远程主机的协议。在 GitHub 场景里，我们主要把它理解成：

**一种让你的电脑能够安全地向 GitHub 证明“你是谁”的身份验证方式。**

你可以把 SSH 理解成一把“数字钥匙”：

1. 你的电脑本地保存一把私钥
2. GitHub 账号中保存对应的公钥
3. 当你向 GitHub 推送代码时，GitHub 会验证这两把钥匙是否匹配
4. 匹配成功，就允许你访问和推送

所以 SSH 的核心作用不是“下载更快”，而是：

**更方便、更稳定地完成身份认证。**

---

## SSH 在 GitHub 协作中的作用

在团队协作里，SSH 主要有下面几个作用。

### 1. 免去频繁输入账号密码

如果使用 HTTPS，有时在 `git push` 时需要输入 GitHub 账号、Token 或走浏览器授权。

如果使用 SSH，配置好之后，通常不需要每次手动输入认证信息。

### 2. 适合长期开发协作

因为你们不是只提交一次，而是会持续地：

- 拉取代码
- 提交代码
- 推送代码
- 同步仓库

这种长期协作场景下，SSH 会更省事。

### 3. 让每个人的电脑拥有自己的访问身份

每个队友都可以在自己的电脑上生成一对 SSH key，并把公钥绑定到自己的 GitHub 账号。

这样 GitHub 能知道是谁在操作仓库。

### 4. 方便和 Git 命令行配合

对于经常使用终端、VS Code、CLion、Keil 配套 Git 插件的人来说，SSH 通常体验更顺。

---

## SSH 和 HTTPS 的区别

### HTTPS 的特点

优点：

1. 不需要提前配置 SSH key
2. 上手最简单
3. 复制仓库地址就能直接克隆

缺点：

1. 推送时代码认证可能稍麻烦
2. 可能需要 Token、网页登录或额外认证步骤

### SSH 的特点

优点：

1. 配好之后长期使用很方便
2. 推送和拉取更顺畅
3. 更适合经常协作开发

缺点：

1. 第一次配置会比 HTTPS 多几步
2. 需要理解公钥和私钥的概念

### 建议

如果只是临时看一下仓库，HTTPS 足够。

如果是要长期参与这个比赛项目，建议配置 SSH。

---

## 公钥和私钥分别是什么

配置 SSH 时，通常会生成两个文件：

1. 公钥
2. 私钥

### 私钥

私钥保存在你自己的电脑里，**绝对不能发给别人**。

它相当于你家的钥匙本体。

### 公钥

公钥可以上传到 GitHub。

它相当于告诉 GitHub：“以后拿着对应私钥的人，就是我这台电脑。”

### 一定要记住

1. 公钥可以上传
2. 私钥不能泄露
3. 私钥丢了或者怀疑泄露，就重新生成一对新的 key

## Windows 上配置 SSH 的完整流程

下面以 Windows 为例说明，适合你们当前环境。

---

## 第一步：检查电脑是否已经有 SSH key

打开 PowerShell 或 Git Bash，输入：

```bash
Get-ChildItem ~/.ssh
```

如果你看到类似文件：

```text
id_rsa
id_rsa.pub
```

或者：

```text
id_ed25519
id_ed25519.pub
```

说明电脑上可能已经有 SSH key 了。

其中：

- 没有 `.pub` 后缀的是私钥
- 有 `.pub` 后缀的是公钥

如果目录不存在或者没有这些文件，也没关系，直接生成新的就行。

---

## 第二步：生成新的 SSH key

现在 GitHub 更推荐使用 `ed25519` 算法。

在终端中输入：

```bash
ssh-keygen -t ed25519 -C "你的GitHub邮箱"
```

例如：

```bash
ssh-keygen -t ed25519 -C "example@qq.com"
```

执行后一般会看到提示：

```text
Enter file in which to save the key (...)
```

这时直接按回车，使用默认路径即可。

然后还会提示你输入密码：

```text
Enter passphrase (empty for no passphrase):
```

这里有两种选择：

### 方式 1：直接回车，不设置密码

优点：

1. 使用方便
2. 每次认证更省事

缺点：

1. 如果电脑本身安全性不足，私钥泄露风险更大

### 方式 2：设置一个 SSH key 密码

优点：

1. 更安全

缺点：

1. 使用时可能需要额外输入密码

对于比赛项目，如果电脑是自己在用、管理比较稳定，可以不设密码；如果更看重安全，可以设一个。

---

## 第三步：确认生成成功

再次执行：

```bash
Get-ChildItem ~/.ssh
```

如果看到：

```text
id_ed25519
id_ed25519.pub
```

说明生成成功了。

---

## 第四步：查看公钥内容

把公钥内容打印出来：

```bash
Get-Content ~/.ssh/id_ed25519.pub
```

你会看到一长串类似下面的内容：

```text
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI...... your_email@example.com
```

这整整一行就是你的公钥。

**复制整行内容，不要漏，不要换行，不要多空格。**

---

## 第五步：把公钥添加到 GitHub

### 1. 登录 GitHub

打开 GitHub 官网并登录自己的账号。

### 2. 进入设置

点击右上角头像，进入：

`Settings`

### 3. 找到 SSH keys 管理页面

进入：

`SSH and GPG keys`

### 4. 新建 SSH key

点击：

`New SSH key`

### 5. 填写信息

一般需要填两个内容：

#### Title

可以写这把 key 对应的电脑名称，例如：

`My Laptop`

或者：

`Dell-Windows`

#### Key

把刚才复制的公钥整行粘贴进去。

### 6. 保存

点击保存后，这台电脑就和你的 GitHub 账号建立了 SSH 身份绑定。

---

## 第六步：测试 SSH 是否配置成功

在终端执行：

```bash
ssh -T git@github.com
```

第一次连接时，可能会看到类似提示：

```text
The authenticity of host 'github.com' can't be established...
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

输入：

```text
yes
```

然后如果配置成功，通常会看到类似信息：

```text
Hi your-github-name! You've successfully authenticated...
```

这就说明 SSH 已经可用了。

---

## 第七步：使用 SSH 地址克隆仓库

配置好之后，就尽量使用 SSH 地址操作仓库。

你们当前仓库的 SSH 地址是：

```text
git@github.com:sysuwilliam/NC-Current-Source.git
```

克隆命令：

```bash
git clone git@github.com:sysuwilliam/NC-Current-Source.git
```

---

## 如果仓库之前是用 HTTPS 克隆的，怎么改成 SSH

如果队友已经用 HTTPS 克隆了仓库，也不用重新下载，直接修改远程地址即可。

先查看当前远程地址：

```bash
git remote -v
```

如果看到的是：

```text
https://github.com/sysuwilliam/NC-Current-Source.git
```

就可以改成 SSH：

```bash
git remote set-url origin git@github.com:sysuwilliam/NC-Current-Source.git
```

然后再次检查：

```bash
git remote -v
```

如果显示的是 SSH 地址，就说明修改成功。

---

## 为什么有时 SSH 还是连不上

下面是几个常见原因。

### 1. 公钥没有加到 GitHub

最常见。你本地虽然生成了 key，但没有把 `.pub` 内容贴到 GitHub 设置里。

### 2. 加错 GitHub 账号

比如你本地生成的 key 上传到了 A 账号，但现在操作的是 B 账号仓库。

### 3. 私钥文件路径不对

有时系统没有自动找到正确的私钥文件。

### 4. 公司、校园网或某些网络限制了 SSH

有些网络环境会拦截 22 端口，导致连 GitHub SSH 失败。

### 5. 仓库没有权限

即使 SSH 配好了，如果你不是这个仓库的协作者，也不能推送。

---

## 常见报错和理解方式

### 报错 1：`Permission denied (publickey)`

含义通常是：

GitHub 没有识别出你的 SSH 身份。

优先检查：

1. 是否真的生成了 SSH key
2. 是否把公钥添加到了 GitHub
3. 是否使用了正确的 GitHub 账号

### 报错 2：`Repository not found`

可能原因：

1. 仓库地址写错了
2. 你没有该仓库的访问权限
3. 你登录的 GitHub 账号不是被邀请的那个账号

### 报错 3：连接超时

可能原因：

1. 网络环境拦截 SSH
2. 当前网络不稳定

这种情况下，可以先临时改用 HTTPS。

---

## 如果有多个 GitHub 账号怎么办

有些同学可能会有：

1. 个人 GitHub 账号
2. 学校或其他项目使用的 GitHub 账号

这时一定要注意：

**SSH key 是绑定到具体 GitHub 账号的。**

如果你把公钥加到了错误的账号上，就会导致当前仓库无法正常推送。

对于新手，最简单的做法是：

1. 先只使用一个 GitHub 账号参与比赛项目
2. 一台电脑先只配置一套常用 SSH key

这样最不容易出错。

---

## SSH 配置完成后，日常怎么使用

一旦 SSH 配好，日常 Git 操作和以前一样，只是认证方式变成了 SSH。

例如：

```bash
git pull origin main
git add .
git commit -m "新增串口调试命令"
git push origin main
```

你不需要每次都重新配置 SSH。

---

## 什么时候可以先不配 SSH

如果队友只是：

- 临时浏览仓库
- 下载一下资料
- 很少提交代码

那可以先用 HTTPS。

但如果要长期参与开发，还是建议尽早把 SSH 配好，后面会省很多事。

---

## 配置流程

如果你想把最短版本直接发给队友，可以发下面这段：

### 最简 SSH 配置步骤

1. 打开 PowerShell
2. 执行：

```bash
ssh-keygen -t ed25519 -C "你的GitHub邮箱"
```

3. 一路回车生成 key
4. 执行：

```bash
Get-Content ~/.ssh/id_ed25519.pub
```

5. 复制输出的一整行公钥
6. 登录 GitHub
7. 进入 `Settings -> SSH and GPG keys`
8. 点击 `New SSH key`
9. 把公钥粘贴进去并保存
10. 回到终端执行：

```bash
ssh -T git@github.com
```

11. 如果提示认证成功，就可以使用 SSH 地址克隆或推送仓库了

---

## 总结

SSH 在 GitHub 协作里的作用，本质上就是：

**让你的电脑用一套安全、稳定、方便的方式证明自己的身份。**

对你们这个比赛项目来说，SSH 的价值主要体现在：

1. 长期协作更方便
2. 推送代码更顺手
3. 每个人都能以自己的身份参与开发
4. 后续多人并行开发时管理更清晰

所以结论很简单：

1. 新手可以先理解 HTTPS 和 SSH 的区别
2. 真正要长期参与开发的队友，建议都配置 SSH
3. 配一次之后，后面会省很多麻烦
